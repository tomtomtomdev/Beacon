## Decisions log

- **2026-07-08** — **Slice 3 shipped (heuristic classifier); decisions during the build:** (1) **Category is read from the job TITLE only, not the description** — the biggest decision, forced by the acceptance spot-check: matching the body tags every posting at an AI company as `ai-ml` (their boilerplate constantly says "LLM/AI") and fintech bodies as `backend` ("go-to-market" → `go`), giving ~50% precision. The title names the role; ambiguous/empty residue is by design the slice-9 LLM's job. **Level** still reads title tokens for seniority + a years-of-experience regex over title+body (years are role-specific, not boilerplate). (2) **Removed bare `ai` and `go` keywords** (misfire on "AI Native" sales titles / "go-to-market"); use phrases (`ai engineer`, `ml engineer`, `ai/ml`) or the specific form (`golang`). (3) **`Classifier` is a port with `HeuristicClassifier` as a (pure) adapter** in `adapters/classify/` per SPEC §7 — it's grouped with its slice-9 LLM sibling under one port even though pure; the value objects `Category`/`Level`/`Classification` are domain. Keyword tables are DATA in `adapters/classify/keywords.py`. (4) **Classification cached by content_hash in the pipeline**: `ingest_source` gained a `Classifier` param and classifies only when `JobRepo.content_hash_for` differs; `upsert` gained an optional `classification` and uses `COALESCE(excluded.categories, jobs.categories)` so an unchanged re-poll keeps its prior classification. (5) **`categories` stored as a sorted comma-joined string** in the existing (nullable) `jobs.categories` column — **NULL = never classified** (backfill picks these up), **"" = classified, nothing matched** (honest empty, skipped). No migration needed (columns existed since 001). (6) **Backfill** = `backfill_classifications` use case + `python -m beacon.classify` CLI (`list_unclassified`/`set_classification` repo methods). (7) **`/jobs` category[]/level[] filters are opt-in** like tiers; category match is comma-delimited membership (`',' || categories || ','` LIKE `%,ios,%`) so a filter for `ios` hits multi-label `ai-ml,ios` without substring bugs. (8) **Frontend**: category/level are DESIGN §2 row-2 pill toggles (URL params `category`/`level`); JobTable gains Category chips + a mono uppercased Level column per DESIGN §tokens; shared `taxonomy.ts` gives FilterBar pills and JobTable chips one label source; added token `--text-chip: #4c5964` (DESIGN category-chip text). (9) **Acceptance spot-check tool is live** (`scripts/spot_check_classifier.py`, 4 verified Greenhouse boards, `--all`/eng-only) — network is fine for manual acceptance, never the suite; every miss it found (space-form "back end"/"front end", java/infra/sre/devops/systems-engineer/javascript/aosp) became a keyword-table row + a parametrized test row.

- **2026-07-07** — **Slice 2 shipped (the moat); decisions/deviations during the build:** (1) **`match_evidence TEXT` column added to `companies` (migration 002)** — SPEC §7's schema omits an evidence column, but SPEC §5.3 / PLAN slice 2 require the MANUAL flag to carry an evidence note + date, and the slice-10 drawer will show which registries matched. One free-text column serves both the fuzzy audit trail (`"UK Skilled Worker; NL KvK …; US 2 certified LCA filings"`) and MANUAL notes (`"MANUAL: listed on relocate.me (flagged 2026-07-07)"`); the schema test was updated to match. (2) **Matching rule = distinctive-token-set EQUALITY, not overlap** (the "Cohere rule"): after dropping SUFFIX/GEO/STRUCTURAL token tables (data in `domain/matching.py`), the remaining sets must be equal — this is what rejects Cohere Health / Notion Capital / Stripe Consulting / Grab+Go while matching Cohere US / Notion Labs / Stripe Payments. STRUCTURAL deliberately includes divisional words seen in real subsidiaries (operations/software/payments/financial/technologies/labs/markets/…) but **excludes** consulting/capital/partners/investments/advisors (the trap generators). Confidence: 1.0 when suffix-only forms already match, 0.9 when geo/structural stripping was needed, None below. (3) **Tier resolution is owned by `refresh_registries`** (single `resolve_tier(text_tier, flags)` precedence fn + a bulk `JobRepo.resolve_registry_tier` that skips explicit-text tiers) — ingest is untouched this slice; a new job for a flagged company gets its tier on the next refresh. (4) **JobDetail "which registries matched + confidence" display deferred to slice 10** with the rest of the drawer (per PLAN UI build note) — slice-2 UI is the sort segmented control + opt-in tier dropdown on the Jobs view (badge already shipped in slice 1). (5) **`registries_meta` table created but not yet populated** — `fetched_at`/`row_count` writing lands in slice 11 alongside the staleness-nag reader that consumes it. (6) **Registry snapshots via `Settings` paths** (`data/registries/`, gitignored; env-overridable) — real exports dropped there for `python -m beacon.refresh`; only trimmed fixtures are committed and the spot-check script reads those. Missing snapshots are skipped, not fatal. (7) **MANUAL preservation:** `refresh_registries` OR-s fuzzy UK/NL/US bits but never clears a prior MANUAL bit; a MANUAL-only company with no fuzzy hit is left entirely intact.
