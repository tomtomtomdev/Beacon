## Decisions log

- **2026-07-08** — **Slice 5.5 shipped (per-job user status); decisions during the build:** (1) **No migration** — `user_status TEXT NOT NULL DEFAULT 'new'` already existed in migration 001 (created ahead for this slice, like `canonical_id`). (2) **`UserStatus` StrEnum in `domain/status.py`** is the single source of truth for valid values (new/seen/hidden/starred) + `DEFAULT_VISIBLE = all − {hidden}`, mirroring the `SponsorTier` pattern; the API query/body params are typed against it so bad values 422 automatically (no hand-rolled validation). (3) **Reset-on-hash-change lives in the upsert SQL** as `user_status = CASE WHEN jobs.content_hash != excluded.content_hash THEN 'new' ELSE jobs.user_status END` — a materially edited posting is genuinely new again; an unchanged re-poll (or classification-only change) keeps the user's decision. Pure-SQL, no extra round-trip. (4) **Status is a property of the canonical row** (the one the list shows). `PATCH /jobs/{id}/status` resolves a duplicate id → its canonical before updating (same resolution as `get_job_detail`) and returns `{id: canonical_id, user_status}`; the `/jobs` `status` filter reads the canonical row's status (list already filters `canonical_id IS NULL`). (5) **`/jobs` `status` param semantics:** absent → default view = everything except `hidden`; `status=all` → no filter (includes hidden); a specific value → only that status. The UI's four views map: New→`status=new`, Starred→`status=starred`, **All→ no param (excludes hidden)**, Hidden→`status=hidden` — so the UI never sends `status=all` (that API affordance exists but is unused by this UI; DESIGN's "All" = all-but-hidden). (6) **Default UI view is New** (the morning scan — the slice's whole point); it's kept out of the URL for clean shared links, other views set `?status=`. (7) **`user_status` exposed on `/jobs` list + detail DTOs** (new read-model field) so the table can render the new-dot, star-fill and hidden-greying without an extra fetch. (8) **Live segment counts deferred to slice 10** — DESIGN §2 wants a live count in each segment (and the nav "Jobs" badge = new-count); implementing now would add 4 count queries and reorder the existing test suite's `fetch` calls. Deferred to the slice-10 nav/drawer UI pass, consistent with the drawer/tier-badge defers. (9) **Row keeps the external-link action** alongside the new star + hide/restore buttons (actions column widened 64→100px) — the "open original" affordance otherwise only returns with the slice-10 drawer CTA, and users need it in the interim.
- **2026-07-08** — **Slice 5 shipped (cross-source dedup); decisions during the build:** (1) **Match key = exact `(company_id, normalized_title, country)` AND simhash(description) within Hamming ≤ 8** (SPEC §5 key 2). The exact title+company+country tuple is the *primary* false-merge guard ("Senior iOS" vs "Senior Android" never collide because titles differ); the simhash only has to reject the rare generic-title collision ("Software Engineer" twice at one company for two unrelated roles). `normalize_title` is deliberately conservative — lowercase + whitespace-collapse only; punctuation differences keep titles distinct (under-merge > over-merge). (2) **simhash hand-rolled in `domain/dedup.py`** (pure, no dep — PLAN allowed a lib but stdlib-first + it's central to correctness): 64-bit Charikar simhash over lowercased alphanumeric tokens weighted by frequency, hashed with **blake2b (deterministic across processes)** not the salted builtin `hash()`. (3) **Threshold 8 chosen on evidence, not to pass a test:** measured on a realistic ~77-token JD — whitespace/punct-only diff = 0 bits, short footer = 1, added sentence = 1, a 15-word footer = 7; a different eng role = 22, a marketing role = 14. 8 sits in the wide 7↔14 gap, biased toward merge (showing the same job twice is the worse sin). **simhash is noisy on very short text** (a 5-word footer on a 28-token doc flips 11 bits) — tests therefore use realistic-length descriptions; the domain boundary test (`test_dedup.py`) documents the near/far separation. (4) **Canonical = lowest id in a cluster** (earliest-inserted, stable/deterministic); clustering is transitive union-find within each group. Canonical row keeps `canonical_id = NULL`; duplicates point at it. `assign_canonicals(rows) -> {id: canonical_id}` is **pure** (fast unit tests, no DB); the `dedupe_jobs` use case just applies the mapping via the repo. **Idempotent** — re-running yields the same links. (5) **Dedup runs as a post-upsert pass over the whole DB** (PLAN task wording), once after `ingest_all` — a Greenhouse job can't be compared to a Lever job until both are persisted. Wired into `beacon.ingest` (prints `dedup groups= duplicates=`). Zero changes to domain from the app side beyond the new pure module; `JobRepo` port gained `list_dedup_rows`/`set_canonical_links`/`get_job_detail`. (6) **`/jobs` list now filters `canonical_id IS NULL`** (canonical-only); existing rows default NULL so no behaviour change for undeduped data. (7) **`GET /jobs/{id}` detail** resolves a duplicate id up to its canonical and returns `description` + `duplicate_sources` (every underlying posting's source/company/url). (8) **No migration** — `canonical_id` column has existed since migration 001. (9) **Frontend scope: backend + contract-types only** — `JobDetail`/`DuplicateSource` mirrored in `types.ts`; the interactive detail *drawer* stays deferred to slice 10 where DESIGN.md owns it (consistent with the slice-2 decision to defer the drawer). The acceptance ("one row in UI") is met by the canonical-only list. *(User was away when asked to confirm this scope fork; chose the no-throwaway option and recorded it here — revisit if a minimal drawer is wanted sooner.)*
